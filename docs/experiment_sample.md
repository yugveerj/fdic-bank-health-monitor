# Experiment analysis: a worked demonstration

This is a method demonstration, not a discovery. It runs the full A/B analysis
pipeline — assignment, sample-ratio check, A/A validation, power math, and a
significance test — against Google's public GA4 e-commerce sample
(`bigquery-public-data.ga4_obfuscated_sample_ecommerce`, Nov 1, 2020 –
Jan 31, 2021). No experiment ever ran on that site as far as this split is
concerned; the one "effect" reported below was injected at a known size to
show the pipeline detects what it is powered to detect. The real,
pre-registered experiment on this project's own GA4 traffic is Phase F
Stage 2. Generated by `uv run python -m analysis.sample_experiment` on
2026-07-05; the numbers reproduce exactly on rerun.

## Hypothesis framing

The stand-in hypothesis mirrors the shape of a real one: *a treatment raises
the per-user purchase conversion rate by 20% relative*. Unit of analysis is
the user (`user_pseudo_id`), metric is users with at least one `purchase`
event divided by all users in the arm, over the full three-month window.
Counting converting users rather than purchase events keeps the unit of
analysis and the unit of randomization the same, which is what makes the
z-test's independence assumption defensible.

## Assignment mechanism

`MOD(ABS(FARM_FINGERPRINT(user_pseudo_id)), 2)` — odd hash lands in
treatment, even in control. Deterministic hash-mod assignment means no
assignment table to store and no drift between runs: the same user is in the
same arm every time the query executes. In a live experiment the same hash
would gate the actual treatment, so exposure and analysis can never disagree.

## Sample-ratio mismatch check

Run before reading any metric: if the split itself is off, every downstream
number is suspect.

| arm | users | share |
|---|---|---|
| control | 135,059 | 49.993% |
| treatment | 135,095 | 50.007% |

Chi-square goodness of fit against 50/50: chi2 = 0.005, p = 0.9448.
No evidence of mismatch; the hash splits this population evenly enough to proceed.

## A/A test

Neither arm was treated, so the conversion rates should differ only by noise.
This is the pipeline's false-positive check: a significant result here would
mean the machinery manufactures effects out of nothing.

| arm | users | converters | conversion |
|---|---|---|---|
| control | 135,059 | 2,218 | 1.6422% |
| treatment | 135,095 | 2,201 | 1.6292% |

Two-proportion z-test: z = -0.267, p = 0.7896 —
null, as it should be.

## Power

With the observed baseline and a 20% relative minimum detectable effect
(two-sided alpha 0.05, power 80%):

| quantity | value |
|---|---|
| baseline conversion (pooled) | 1.6357% |
| relative MDE | 20% |
| required users per arm | 25,916 |
| available users per arm | 135,059 |
| powered | yes |

Three months of this sample's traffic is enough to detect a 20% relative lift at the planned error rates.

## Synthetic-lift test

A 20% relative lift is injected into the treatment arm with
`apply_synthetic_lift` (seed 20260705): each non-converter converts with the
probability that yields +20% in expectation, drawn once from a seeded
generator. The true effect size is known by construction, so this asks one
question only — does the pipeline detect an effect of exactly the size the
power calculation promised it could?

| arm | users | converters | conversion |
|---|---|---|---|
| control | 135,059 | 2,218 | 1.6422% |
| treatment (+20% injected) | 135,095 | 2,685 | 1.9875% |

Two-proportion z-test: z = +6.721, p = 1.80e-11 —
detected, consistent with the power analysis.

## Limitations

The detected effect exists because this script put it there; nothing here says
anything about any real treatment. The traffic is Google's obfuscated
demo-store sample, not this project's — its baseline conversion and volume
transfer no conclusions to our dashboard's audience, and its
`traffic_source` fields are user-scoped first-touch attribution, unusable for
session-level segmentation anyway. The A/A test is a single draw, not a
false-positive-rate estimate; it can fail 5% of the time by design. The real
test — pre-registered hypothesis, MDE, and stopping rule written down before
launch — runs on this project's own GA4 export in Phase F Stage 2.
